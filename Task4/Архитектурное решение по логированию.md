# Архитектурное решение по логированию

## 1. Какие логи собирать и из каких систем
Сбор логов нужен из всех сервисов, где создается или меняется заказ:
- Shop API
- CRM API
- MES API
- RabbitMQ
- Хранилище 3D файлов
- Базы данных Shop DB и MES DB

### INFO логи, которые нужны всегда
1. Создание заказа
   - время, order_id, user_id, источник (web или api).
2. Загрузка 3D файла
   - время, order_id, user_id, имя файла.
3. Запуск расчета цены
   - время, order_id, service, параметры модели.
4. Результат расчета цены
   - время, order_id, цена, длительность.
5. Смена статуса заказа
   - время, order_id, старый статус, новый статус, кто инициировал.
6. Отправка сообщения в очередь
   - время, order_id, exchange, routing_key.
7. Получение сообщения из очереди
   - время, order_id, queue, consumer.
8. Создание заказа через API партнера
   - время, order_id, api_key.
9. Ошибка обработки 3D файла
   - время, order_id, error_code.
10. Ручные действия оператора
   - время, order_id, operator_id, действие.

### Другие уровни логирования
- WARN: нестандартные ситуации, например повторное сообщение или подозрительно долгий расчет.
- ERROR: ошибки, из-за которых заказ не может продолжить путь.
- DEBUG: только временно, при расследовании инцидентов, ограничено по времени.

## 2. Мотивация
Логи позволяют понять, что произошло с заказом, без опроса клиента. Это уменьшает нагрузку поддержки и ускоряет поиск причин.
Метрики, на которые влияет логирование:
- Среднее время разбора инцидента.
- Количество обращений в поддержку по заказам.
- Доля заказов со сбоями на каждом этапе.
- Уровень соблюдения сроков производства.

### Где начинать логирование и трейсинг в первую очередь
В первую очередь Shop API, MES API и RabbitMQ. Именно там возникают задержки и потери заказов, и это влияет на всех клиентов.

## 3. Предлагаемое решение
### Технологии
- Агент сбора логов, например Filebeat + Logstash.
- Хранилище и поиск логов, например Elastic.
- Интерфейс для поиска и дашбордов, например Kibana.

### Компоненты и связи
- Каждый сервис пишет структурированные логи в формате JSON.
- Агент собирает логи и отправляет в хранилище.
- Команда работает с логами через интерфейс.
- Схема с компонентами логгирования: [logging.drawio](/Task4/logging.drawio).

### Безопасность
- Доступ к логам только у поддержки, разработчиков и админов.
- Ролевая модель доступа и аудит запросов.
- Маскирование персональных данных и API ключей.
- Ограничение внешнего доступа, только внутренняя сеть.

### Хранение
- Отдельный индекс на каждый сервис.
- Хранение 30 дней для горячих логов, 180 дней для архивных.
- Ротация по размеру и времени.

## 4. Переход от сбора к анализу
- Настроить алерты на рост ошибок и необычный всплеск событий.
- Добавить поиск аномалий по количеству заказов и расчетов.
- Выделить дашборды для поддержки и разработки.

## 5. Выбор технологии логирования

| Критерий | ELK Stack (Elastic) | OpenSearch | Splunk | Grafana Loki | Graylog |
|---|---|---|---|---|---|
| 1. Лицензия | Проприетарная (ELv2/SSPL) | Полностью Open Source (Apache 2.0) | Коммерческая (Proprietary) | Open Source (AGPLv3) | Open Source (SSPL) / Enterprise |
| 2. Потребление ресурсов | Высокое (требует много RAM для индексов) | Высокое (аналогично ELK) | Среднее/Высокое (оптимизировано, но платно) | Очень низкое (индексирует только метаданные) | Среднее (легче Logstash, но нужен ES/OpenSearch) |
| 3. Полнотекстовый поиск | Отличный (на базе Lucene) | Отличный (форк Elasticsearch) | Мощный проприетарный движок | Ограничен (эффективен только с метками) | Хороший (упрощенный интерфейс над ES) |
| 4. Сложность масштабирования | Высокая (требует настройки шардов и JVM) | Высокая (аналогично ELK) | Средняя (хорошие встроенные инструменты) | Низкая (Cloud-native дизайн) | Средняя (проще, чем чистый ELK) |
| 5. Визуализация | Kibana (универсальная) | OpenSearch Dashboards | Встроенные Dashboards (очень гибкие) | Grafana (стандарт для метрик и логов) | Собственный GUI (упор на логи) |
| 6. Основной фокус | Поиск, аналитика, Observability | AWS-совместимость, Open Source поиск | Security (SIEM), бизнес-аналитика | Kubernetes, экономия на хранении | Удобство «из коробки», администрирование |
| 7. Модель оплаты | Free / Tiered (за функции/облако) | Бесплатно (Self-hosted) / Managed (AWS) | За объем данных (Ingest) или нагрузку | Бесплатно / Managed (Grafana Cloud) | Бесплатно до 2ГБ/день (Enterprise далее) |

Выбор сделан в пользу ELK - так как с этой технологией я знаком, но также очень хороша альтернатива в виде OpenSearch.
