# Архитектурное решение по кешированию

## 1. Анализ
Проблемы MES :  
Операторы жалуются на низкую скорость работы со страницей.  
Новых клиентов не устраивает скорость выполнения заказа.  

Значит кешируем запросы к MES.

## 2. Мотивация
Кеширование нужно, чтобы:
- операторы могли быстрее загружать страницы  
- нагрузку на MES DB и MES API была уменьшена
- уменьшить задержки в производственном процессе.

#### Элементы системы для кеширования: 
- **Данные о производственных процессах:** часто используемые данные о текущих и планируемых производственных процессах. 
- **Информация о запасах:** данные о текущих запасах и их доступности, которые часто запрашиваются для принятия решений. 
- **Информация о заказах:** данные о текущих заказах и их статусах, которые часто запрашиваются в процессе работы. 
- **Результаты вычислений:** промежуточные результаты вычислений, которые могут быть повторно использованы без необходимости пересчёта.

Кеширование этих данных позволит сократить время на их загрузку и обработку, что приведёт к повышению общей производительности системы и улучшению пользовательского опыта.

## 3. Предлагаемое решение
### Тип кеширования
Предлагаю серверное кеширование для ускорения работы системы и снижения нагрузки на MES, особенно при расчёте стоимости производства изделий.

### Почему серверное кеширование: 
1. **Снижение нагрузки на базу данных и сервер:** расчёты стоимости изделий могут занимать значительное время, особенно для сложных моделей. Кеширование результатов этих расчётов позволит ускорить обработку повторных запросов. 
2. **Ускорение загрузки данных:** кеширование данных, которые часто запрашиваются (например, список заказов на дашборде), поможет ускорить загрузку страниц и улучшить взаимодействие пользователей с системой.

#### Выбор паттерна серверного кеширования
- **Cache-Aside:** этот паттерн подходит для сценариев, где данные загружаются в кеш по запросу, если их нет в кеше. Это позволит минимизировать задержки при первом запросе и ускорить последующие запросы за счёт использования кешированных данных. 
- **Write-Through:** может быть менее подходящим, так как при каждом изменении данных потребуется записывать их в кеш и базу данных, что может снизить производительность при частых изменениях. 
- **Refresh-Ahead:** этот паттерн требует предварительного планирования и загрузки данных в кеш перед ожидаемым запросом, что может быть сложно реализовать в условиях динамической нагрузки и непредсказуемых запросов.

Таким образом, **Cache-Aside** кажется наиболее подходящим паттерном для «Александрита», так как он позволяет гибко управлять кешированием данных и ускорять обработку запросов без необходимости предварительной загрузки данных в кеш.

### Технологии
- Redis как кеш.
- MES API как единственная точка доступа к кешу.

### Диаграмма последовательности
Схема: [sequence_caching.drawio](/Task5/sequence_caching.drawio).

### Инвалидация кеша
Используем комбинированную стратегию:
- По ключу: при смене статуса инвалидируем кеш по затронутым спискам.
- По времени: TTL 30-60 секунд, чтобы не держать устаревшие данные.

Почему не только по времени:
- слишком много устаревших данных, операторы не видят новые заказы.
Почему не только по ключу:
- риск, что часть кейсов не учтена, поэтому TTL как страховка.

### Сравнение стратегий инвалидации
| Стратегия | Плюсы | Минусы | Когда подходит |
| --- | --- | --- | --- |
| По ключу | быстрые актуальные данные | требует точного учета всех изменений | списки заказов в MES |
| По времени | просто внедрять | возможна устаревшая информация | справочники |
| Комбинированная | баланс актуальности и надежности | сложнее в реализации | основные списки |

## 4. Дополнительные варианты
### Вариант 1: Cache-Aside + Redis (основной)
Плюсы: быстро внедряется, гибкая инвалидация.
Минусы: нужно следить за ключами и TTL.

### Вариант 2: Материализованные представления в БД
Плюсы: снижает сложность кеша.
Минусы: нагрузка на БД, сложные миграции.

### Итог
Основной вариант — Cache-Aside с Redis, потому что он быстрее дает эффект и проще масштабируется.
